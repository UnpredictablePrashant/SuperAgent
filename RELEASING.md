# Release Checklist

Use this checklist before tagging `v*` releases.

The goal is simple: publish only what the repository can currently prove.

## 1. Scope The Release

- confirm the release theme in one sentence
- list the primary user-visible changes
- identify any compatibility notes, migrations, or setup changes
- remove claims from release notes that are not covered by code or verification

## 2. Verify The Repo

Run:

```bash
python scripts/verify.py ci --strict-docker
```

If Docker is not available locally, confirm the CI run completed successfully before publishing.

Minimum release bar:

- package builds successfully
- `compile`, `unit`, `smoke`, and `docs` pass
- Docker Compose config validation passes
- release docs links resolve

## 3. Review Public-Facing Docs

Check these files for drift:

- [README.md](README.md)
- [docs/index.md](docs/index.md)
- [docs/quickstart.md](docs/quickstart.md)
- [docs/install.md](docs/install.md)
- [docs/verification.md](docs/verification.md)
- [docs/core_workflows.md](docs/core_workflows.md)
- [CHANGELOG.md](CHANGELOG.md)

Confirm:

- the recommended entry workflows are still accurate
- install and verify commands match the repo
- setup requirements are concrete
- feature status language matches reality

## 4. Prepare Release Notes

Include:

- what Kendr is today
- the strongest workflows improved in this release
- any setup or compatibility changes
- any verification or CI changes
- any known gaps that still remain

Prefer a few strong examples over long feature inventories.

## 5. Package And Tag

- update [CHANGELOG.md](CHANGELOG.md)
- commit the release notes or release-related doc updates
- create and push a tag in the form `vX.Y.Z`
- confirm the GitHub release workflow uploaded `dist/*`

## 6. Post-Release Checks

- install from the published artifact in a clean environment when practical
- sanity check `kendr --help`
- sanity check `kendr setup status`
- sanity check one primary workflow command from [docs/core_workflows.md](docs/core_workflows.md)
- confirm the GitHub release page has a clear summary, not only autogenerated notes

## 7. If Something Is Missing

Do not release around missing basics such as:

- broken install path
- failing default verifier
- outdated README setup commands
- undocumented compatibility changes
- public claims that exceed current verification
